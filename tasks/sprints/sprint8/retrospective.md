# Sprint 8 Retrospective

日期：2026-03-21
范围：基于 Sprint 8 review 结果，对本轮迭代进行复盘与改进规划。

## 1. 本轮目标回顾
Sprint 8 目标是完成通道与算子目录服务能力建设，并打通从 Web 管理到 Scheduler 执行再到持久化与预览的闭环。

从结果看，本轮不仅交付了目标能力，还完成了关键架构收敛：
1. 注册中心从 `IDataFrameRegistry` 统一为 `IChannelRegistry`。
2. 插件定位从 `BuiltinPlugin` 明确收敛为 `CatalogPlugin`。
3. Web 端通道管理交互从“并列分区”升级为“类型切换式侧边栏”。
4. 预览链路从一次性加载改为分页加载，显著提升稳定性。
5. SQL 执行链路补齐大小写兼容，修复线上可复现故障。

## 2. 做得好的地方
1. **问题闭环速度快**：从用户反馈到定位根因、修复、回归验证形成了短闭环，避免问题长期堆积。
2. **架构方向统一**：及时放弃过窄抽象（`IDataFrameRegistry`），避免后续扩展成本持续放大。
3. **命名治理有效**：插件、目录、动态库命名统一到 `catalog`，系统语义更清晰。
4. **性能改进直接见效**：预览分页后，100+ 列数据场景的浏览器卡顿得到缓解。
5. **验证纪律到位**：关键改动均经过构建与端到端测试确认（`test_builtin`、`test_scheduler_e2e`）。

## 3. 暴露出的不足
1. **早期设计与实现偏差**：文档先行定义了 `IDataFrameRegistry`，实现后期才发现抽象层级不对。
2. **大小写兼容覆盖不足**：SQL `FROM` 替换、目录匹配等路径初期未统一考虑大小写输入。
3. **文档同步滞后**：代码完成后，`planning/design/backlog/lessons` 一度与实际命名和接口不一致。
4. **命名收敛未一次完成**：虽已迁移到 `CatalogPlugin`，但测试目录名仍有历史痕迹（`test_builtin`）。

## 4. 根因分析
1. **接口边界定义偏实现导向**：以“当前只有 DataFrame”驱动接口命名，弱化了“统一通道目录”的领域边界。
2. **兼容性用例前置不足**：缺少“大小写变体输入”作为标准回归用例模板。
3. **变更管理粒度偏粗**：命名、目录、接口、文档的演进没有作为一个“原子迁移包”一次推进。

## 5. 具体改进动作（下个 Sprint 执行）
1. **接口治理**：新增“接口命名检查清单”，要求命名优先体现领域抽象层级而非当前实现类型。
2. **测试增强**：为 Scheduler 增加大小写兼容回归样例（`from/FROM`、`dataframe/DataFrame`、`builtin/BUILTIN`）。
3. **文档门禁**：涉及接口/命名变更时，PR 必须同步 `planning + design + review + backlog`。
4. **命名收尾**：将 `src/tests/test_builtin/` 评估迁移为 `src/tests/test_catalog/`（非阻塞，建议尽快完成）。
5. **性能基线**：为 DataFrame 预览建立基准场景（高列数、高行数）并记录分页策略阈值。

## 6. 经验沉淀
1. **领域概念优先于实现细节**：通道目录服务应该以 `Channel` 为一级概念，而不是具体子类型。
2. **“用户输入不规范”是默认场景**：SQL 与通道标识的大小写兼容必须系统化处理。
3. **命名即架构**：`catalog` 比 `builtin` 更能传达职责边界，减少团队沟通成本。
4. **性能问题优先做结构性修复**：分页比局部优化更稳健、收益更直接。

## 7. 总结
Sprint 8 的核心成果不仅是功能交付，更是一次架构去债与语义收敛：
- 接口从特化走向统一；
- 命名从模糊走向明确；
- 交互从可用走向可扩展；
- 修复从点状走向系统化（兼容性 + 文档 + 测试）。

整体结论：本轮迭代达成目标，并为后续多类型通道扩展建立了正确基础。
