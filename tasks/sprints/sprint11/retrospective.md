# Sprint 11 Retrospective

日期：2026-03-28  
范围：基于 Sprint 11 交付与 review 结果，复盘 PostgreSQL 通道、算子三类型统一、C++ 插件管理链路与前端改造过程。

## 1. 本轮目标回顾
Sprint 11 的核心目标是把“算子管理”从双类型扩展到三类型（`builtin/python/cpp`），并补齐 PostgreSQL 通道能力，同时保证 `/operators/*` 统一入口与端到端可测试性。

结果：
1. 目标主线已打通：后端统一 URI、前端分类型展示、C++ 插件上传/激活/去激活/删除/详情链路可用。
2. PostgreSQL 类型已补入 Web 通道管理入口，功能漏项得到纠正。
3. 本轮新增修复完成：Bridge 并发读写竞态修复、前端静态产物目录膨胀修复、路由懒加载拆包落地。

## 2. 做得好的地方
1. 需求澄清充分：围绕 URI 统一、`category` 语义收敛、`plugin_id` 唯一标识等关键点进行了持续讨论并形成一致方案。
2. 架构方向正确：`CatalogPlugin` 统一 `/operators/*` 入口，`BinAddonHost` 专注 C++ 插件生命周期，职责边界清晰。
3. 问题闭环及时：发现前端产物膨胀后快速定位到 CMake 同步策略并修复，目录体积从历史累积状态回到合理水平。
4. 质量意识提升：对并发风险进行了专项审视并补上 Bridge 并发回归测试，避免了隐蔽的运行时不确定性。

## 3. 暴露出的不足
1. 设计与实现出现过偏差：曾出现“实现先行、偏离既定设计”的情况，后续需要返工收敛。
2. 漏项发现偏后：PostgreSQL 类型在 Web 通道页未同步接入，说明“跨层同步检查”做得不够前置。
3. 并发与一致性风险仍有存量：BinAddon 仍有若干事务原子性、锁粒度、失败回滚与测试覆盖问题暂缓处理。
4. 前端性能治理仍在中途：已完成路由拆包，但主入口 chunk 仍偏大，尚未完成 UI 库按需与进一步分包。

## 4. 根因分析
1. 交付压力下的实现策略偏向“最小改动快完成”，与“设计一致性优先”原则存在冲突。
2. 缺少“设计到实现映射清单”门禁，导致多模块联动需求在局部实现时易遗漏。
3. 稳定性测试节奏后置：复杂链路（插件激活失败回滚、并发冲突）更多在 review 阶段暴露，而非开发中期。

## 5. 下个 Sprint 的改进动作
1. 建立“设计一致性门禁”：
   - 实现前输出“设计条目 -> 代码点位”映射清单；
   - 合并前逐项核对，不满足则不收口。
2. 建立“跨层变更检查清单”：
   - 需求涉及类型/枚举扩展时，必须同步检查 `API -> Catalog -> Scheduler -> Web -> 前端` 五层。
3. 优先清理 P1 存量风险：
   - BinAddon 激活事务化与失败回滚；
   - Deactivate/Delete 强一致；
   - 锁粒度与锁顺序收敛；
   - 失败注入与并发回归测试补齐。
4. 前端继续做第二阶段治理：
   - Element Plus 按需引入；
   - Operators 页面继续拆分；
   - 增加 bundle 体积门禁，防止回弹。

## 6. 经验沉淀
1. 统一入口与分层委派是可扩展路径：`/operators/*` 统一入口 + 类型分支委派，能兼顾一致性与扩展性。
2. 设计不是参考而是约束：实现如果偏离设计，短期看“快”，中期一定以返工成本偿还。
3. 漏项通常发生在“跨层接口变更”而非单模块逻辑，必须用清单化方法防遗漏。
4. 并发与一致性问题不应依赖上线后观察，必须在迭代内通过专项测试前置收敛。

## 7. 总结
Sprint 11 总体达成目标，且在关键风险点完成了多项补强（并发修复、体积治理第一阶段）。  
本轮的主要教训是：在复杂多模块迭代中，必须把“设计一致性门禁 + 跨层检查清单 + 稳定性测试前置”作为同等重要的交付内容。  
下一阶段应以“先清存量风险，再扩展能力”为主策略推进。
