## 设计评审意见（历史）

### 问题 1：单源/多源执行入口分裂风险

- **意见**：不要维护两套算子执行入口；单源是多源的特例，统一走多输入接口。
- **风险**：若保留双入口，后续算子能力扩展会产生行为不一致和维护成本倍增。
- **修正**：`ExecuteWithOperatorChain(Span<IChannel*>)` 作为唯一算子链入口；`inputs.size()==1` 即单源特例。
- **状态**：已采纳并实现。

### 问题 2：“首算子”语义歧义

- **意见**：“首算子”应定义在**单条 SQL 语句级**，而非整个多 SQL Task 级。
- **风险**：Task 级定义会导致 `sqls` 链路中语义冲突，尤其在多段中间通道场景。
- **修正**：设计文档明确 `operators[0]` 为该 SQL 首算子；Task 级不存在唯一首算子。
- **状态**：已采纳并落文。

### 问题 3：多源无算子语义不确定

- **意见**：`FROM a,b` 必须配合 `USING` 算子；否则无法确定合并/对齐语义。
- **风险**：默认拼接策略会制造隐式行为，导致结果不可预期。
- **修正**：入口校验“多源必须有算子”，否则直接 `BAD_REQUEST`。
- **状态**：已采纳并实现。

### 问题 4：多源 V1 能力边界不清

- **意见**：Sprint 10 先收敛能力边界：多源仅支持 `dataframe.*`，且先禁用多源 `WHERE`。
- **风险**：一次性支持数据库多源+过滤会放大适配复杂度和故障面。
- **修正**：在调度入口增加硬约束与明确错误信息。
- **状态**：已采纳并实现。

### 问题 5：多 SQL 中间通道清理策略不确定

- **意见**：清理必须按“任务内显式创建的中间 `dataframe.*` 目标集合”执行，排除最后输出通道。
- **风险**：基于快照或模糊规则清理会误删并发任务通道。
- **修正**：执行前收集非最后一条 SQL 的 `dest`，任务终态后定向清理。
- **状态**：已采纳并实现。
